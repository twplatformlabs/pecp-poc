<objective>
Wire the integration framework into the live API: register integrations during FastAPI
lifespan startup, add `BackgroundTasks` parameters to `create_team` and `create_project`,
build snapshot dataclasses from ORM objects inside the request scope, and schedule
`fire_integrations` AFTER `await session.commit()` so the commit-before-hook invariant
holds in every code path. Cover INTG-02 success criterion 1 (hooks fire after creation),
success criterion 2 (failing integration does not block the primary 201 response), and
ROADMAP success criterion 4 (DB row exists when the hook runs). Extend the unit-level
test_registry.py from Plan 01 with HTTP-level assertions, and add a new
test_commit_ordering.py that proves the invariant by issuing a SELECT inside the hook.

Purpose: Lock the commit-before-hook structural invariant into both route handlers
before Phase 8 introduces real GitHub API calls. After Plan 02, Phase 8 only adds the
`GitHubIntegration` class and the conditional `INTEGRATION_REGISTRY.append(...)` line —
no further wiring changes.

Output: Modified `teams.py`, `projects.py`, `main.py`; new `test_commit_ordering.py`;
extended `test_registry.py`; full suite green.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP client → POST /teams, POST /projects | External request body crosses into route handler; validated by Pydantic before any DB write |
| route handler → BackgroundTasks queue | Snapshot dataclass crosses into background task scope — ORM session is closed by the time the task runs |
| BackgroundTasks → fire_integrations → registered integration | Hook code executes with snapshot data only; no DB session, no request context |
| FastAPI lifespan → os.environ | GITHUB_PAT / GITHUB_ORG read once at startup via IntegrationConfig (pydantic-settings) |
