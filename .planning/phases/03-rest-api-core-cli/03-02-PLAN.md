## Phase Goal

**As a** platform engineer using PECP, **I want to** apply a YAML resource spec, watch the server accept it with 202, poll `/resources/{id}` and see the status transition to `ready`, and append PE notes that show up on subsequent reads, **so that** the full control-plane contract (idempotent submit, async dispatch, append-only notes log, hard delete) is reachable via HTTP before any new CLI surface lands.

<objective>
This plan delivers the server-side vertical slice for Phase 3. A real user (or HTTP client) can now apply a YAML resource, watch the Dispatcher drive it through pending→provisioning→ready entirely via REST, append PE notes, and delete resources — all the Wave 0 RED tests from Plan 01 turn green here. The CLI vertical slice in Wave 3 will then wrap these endpoints.

Purpose: Implement CTRL-01 (POST returns 202 with id), CTRL-02 (BackgroundTasks triggers dispatch with fresh session), CTRL-03 (idempotency on (team, kind, name)), and CTRL-04 (append-only notes). Resolves Pitfall 1 (BackgroundTasks session lifetime) and Pitfall 5 (NoteCreate Pydantic model for 422 on bad body).

Output: A single rewritten `src/pecp/api/routes/resources.py` containing all 5 routes (existing GET list extended with kind filter, existing POST extended with idempotency + BackgroundTasks, new GET-by-id, new DELETE, new POST notes) plus the `_dispatch_with_session` wrapper and `NoteCreate` Pydantic model.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP client → FastAPI route | Untrusted YAML body; untrusted JSON body for notes; untrusted resource_id in path |
| Route handler → BackgroundTasks | Request session lifetime ends before background task runs — must NOT share |
| Route handler → Dispatcher (BackgroundTasks) | resource_id crosses async boundary; session ownership transfers to background coroutine |
| Caller knowing a resource_id → DELETE on cross-team resource | Without team-verify the API leaks delete authority across teams |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-3-02-01 | Tampering | POST /resources idempotency | mitigate | Lookup-before-insert with `(team, kind, name)` (D-08, D-11); catch `IntegrityError` from race window and re-fetch existing row; DB-level `uq_resource_team_kind_name` (Plan 01) is the defense-in-depth backstop |
| T-3-02-02 | Denial of Service | BackgroundTasks session lifetime | mitigate | `_dispatch_with_session` opens its own `async with AsyncSessionLocal()` — request session is never passed to the background task (Pitfall 1) |
| T-3-02-03 | Tampering | DELETE /resources/{id} cross-team | mitigate | DELETE route requires `team` query param (400 if absent), re-verifies `record.team == team` and returns 404 (NOT 403) on mismatch to avoid leaking existence (Assumption A5) |
| T-3-02-04 | Denial of Service | POST /resources/{id}/notes bad body | mitigate | `NoteCreate(BaseModel)` with required `text: str` — FastAPI returns clean 422 instead of a 500 from a KeyError (Pitfall 5) |
| T-3-02-05 | Information Disclosure | GET /resources/{id} response shape | accept | Returns provider_metadata and activity_log alongside notes (Open Question 3). All values come from the same team's own resource; no cross-team leakage. Useful for debugging |
| T-3-02-06 | Tampering | YAML parsing | mitigate | `yaml.safe_load` only (existing T-01-01 invariant — preserved, not weakened) |
| T-3-02-SC | Tampering | No new packages | accept | This plan introduces zero new dependencies. No legitimacy checkpoint required |
</threat_model>
