## Phase Goal

**As a** developer using PECP, **I want to** run `pecp apply -f lambda.yaml`, watch the server accept 202, then run `pecp status PECPLambda my-fn --team platform` and see the resource transition from `pending` to `ready` in subsequent calls — entirely from the terminal against a live server, **so that** I have a real platform engineering workflow without ever touching the API directly.

<objective>
This plan delivers the CLI vertical slice that completes Phase 3's ROADMAP user story. After this plan, a real human can drive the full ROADMAP narrative: `pecp apply`, `pecp get`, `pecp status` (sees notes), `pecp delete` — all from the terminal against a live server, with status color-coded.

Purpose: Implement CLI-01 (apply idempotency-aware output), CLI-02 (get with Rich table + status badges), CLI-03 (delete with team safety), CLI-04 (status with table + notes block — `--watch` is deferred per Deferred Ideas), and confirm CLI-11 URL resolution for the new commands.

Output: A single extended `src/pecp/cli/main.py` with three new Typer commands (`get`, `status`, `delete`), the `status_badge` helper + `STATUS_COLORS` map, and a small `_resolve_base_url` helper to deduplicate URL resolution across all four commands.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User shell → CLI args | Untrusted typer arguments (kind, name, team) flow into URL params; httpx parameterizes safely |
| CLI → REST API | `--api-url` is user-controllable; default is localhost; no auth in PoC |
| CLI → DELETE without `team` param | Without team verification on the client side, a typo could DELETE the wrong team's resource (server still 404s, but CLI ergonomics matter) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-3-03-01 | Tampering | pecp delete cross-team | mitigate | CLI ALWAYS passes `params={"team": team}` on httpx.delete; server (Plan 02) returns 404 on mismatch. List lookup also filters by team so the wrong-team id is never even resolved (Assumption A5, defense in depth) |
| T-3-03-02 | Information Disclosure | pecp status notes block | accept | Notes are owned by the same team that owns the resource; no cross-team leak. Author shown as stub-user in PoC; real identity awaits AUTH-V2-01 |
| T-3-03-03 | Denial of Service | Network errors hang the CLI | mitigate | All httpx calls set `timeout=10.0` (matches existing `apply` pattern); RequestError caught and exits with code 1 + red message |
| T-3-03-04 | Spoofing | --api-url points to attacker server | accept | PoC has no auth and no secrets; URL flexibility is required for CLI-11. Documented constraint, not a fixable threat in Phase 3 |
| T-3-03-SC | Tampering | No new packages | accept | Phase 3 introduces zero new dependencies. No legitimacy checkpoint required |
</threat_model>
